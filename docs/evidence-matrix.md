# Evidence Matrix

This matrix maps the course requirements to concrete first-party evidence in this repository. It is intended for the final report, video planning, and reviewer navigation.

## Requirement Mapping

| Requirement | Evidence | How to verify | Status |
|---|---|---|---|
| Complete source code | `packages/runtime/`, `packages/opencode-adapter/`, `evaluation/` | Inspect first-party files outside `tmp/` | Present |
| Setup instructions | `README.md` | Read Requirements, Scripts, and Validation Evidence sections | Present |
| Testing scripts | `package.json` | Run `npm test` | Present |
| Example outputs | `.runtime-data/opencode-validation/<scenario-id>/<run-id>/` | Inspect `prompt.md`, `artifacts/handoffs/*.md`, `validation-result.json`, and new `recall-log.json` outputs from fresh preparations | Present |
| Knowledge base for RAG | SQLite runtime state generated by `MemoryRuntime` | Run scenario preparation and inspect generated `memory.sqlite` plus `recall-log.json` | Present |
| Prompt Engineering | `packages/opencode-adapter/src/index.ts`, `packages/runtime/src/session-summary.ts`, `evaluation/opencode-scenario.ts` | Inspect prompt document, handoff markdown, and scenario-specific instructions | Present |
| RAG retrieval path | `packages/runtime/src/storage.ts`, `packages/runtime/src/retrieval.ts`, `packages/runtime/src/context-assembly.ts` | Run `npm test` and inspect `ContextBundle.recallLog` for keyword/tag/kind/FTS reasons | Present |

| Performance / result metrics | `evaluation/project-rule.ts`, `.runtime-data/opencode-validation/**/validation-result.json` | Compare baseline vs memory-assisted output and strict OpenCode validation records | Present |

## Verification Commands

```bash
npm test
npm run evaluate:project-rule
npm run prepare:opencode-scenario -- project-rule
npm run prepare:opencode-scenario -- interrupted-task
npm run prepare:opencode-scenario -- architecture-rationale
npm run prepare:opencode-scenario -- decision-update
```

Real OpenCode validation additionally requires a working local OpenCode installation and configured provider:

```bash
npm run validate:opencode-scenario -- project-rule 4111
npm run validate:opencode-scenario -- interrupted-task 4112
npm run validate:opencode-scenario -- architecture-rationale 4113
npm run validate:opencode-scenario -- decision-update 4114
```

## Evidence Quality Notes

- `recall-log.json` is separate from `prompt.md` so the model prompt remains clean while the report still gets retrieval evidence.
- `validation-result.json` proves the observed OpenCode output matched scenario-specific rules.
- `docs/final-delivery-checklist.md` lists the latest prepared artifact directories to use for final report screenshots.
- Older failed validation runs may remain under `.runtime-data/`; use the latest passing run for final screenshots and report excerpts.
- `tmp/` contains reference repositories and should not be cited as first-party implementation.
